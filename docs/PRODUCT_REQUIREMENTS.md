# POD 商品图批量处理系统需求文档

## 1. 文档目的

本文档用于沉淀 POD 商品图批量处理系统的产品需求，方便产品、运营、设计和开发持续补充需求、确认优先级、编写验收标准。

后续新增需求建议直接按本文档的“需求卡片模板”补充，避免只描述功能名称而缺少业务场景、输入输出、边界条件和验收标准。

## 2. 产品定位

POD 商品图批量处理系统是一个内部使用的自动化后台，用于提升 POD 商品从素材图片到可导出上架资料的处理效率。

核心流程：

1. 上传原始图片。
2. 管理素材库。
3. 批量处理图片尺寸、抠图、印花提取。
4. 使用固定套图模板生成商品套图。
5. 使用 AI 图片工作台生成或提取印花素材。
6. 管理商品草稿。
7. 导出 Excel 和图片 ZIP。

## 3. 目标用户

| 用户角色 | 主要目标 | 典型操作 |
| --- | --- | --- |
| POD 运营人员 | 快速批量生成商品图和上架资料 | 上传图片、生成套图、编辑商品草稿、导出文件 |
| 图片处理人员 | 批量清理和处理素材图 | 批量改尺寸、抠图、印花提取、查看失败任务 |
| 产品/管理员 | 配置流程和检查数据质量 | 维护套图模板、检查任务、确认导出结果 |

## 6. 需求状态定义

| 状态 | 含义 |
| --- | --- |
| 待评审 | 产品提出，尚未确认是否做 |
| 已确认 | 业务价值和范围已确认 |
| 设计中 | 需要补充交互、文案、字段或流程 |
| 待开发 | 已具备开发条件 |
| 开发中 | 开发正在实现 |
| 待验收 | 开发完成，等待产品或业务验收 |
| 已完成 | 验收通过 |
| 暂缓 | 暂不做，保留记录 |

## 7. 优先级定义

| 优先级 | 含义 | 示例 |
| --- | --- | --- |
| P0 | 阻塞核心流程，必须立即处理 | 上传失败、导出失败、任务无法执行 |
| P1 | 影响主要效率，应优先安排 | 批量失败重试、草稿搜索、模板删除 |
| P2 | 提升体验或减少人工操作 | 批量筛选增强、预览优化 |
| P3 | 可选优化或长期能力 | 多平台发布、复杂权限 |

### 8.9 AI 图片工作台

当前能力：

- 支持文生图、图生图 AI 提取印花、印花图换底和 AI 生成印花。
- API Key 只在后端或服务端配置中读取。
- 支持 Seedream 图片模型配置。
- 支持常用提示词模板，用户可以直接编辑提示词。
- 图生图默认面向服装印花提取，提示词会限制衣服、人物、背景、布料纹理、褶皱、阴影、低清晰度、乱码文字、水印、边框、裁切和重复图案等干扰。
- 印花图换底使用前端本地处理，输出透明底 PNG，不依赖外部 API。
- 生成图片保存到素材库，便于后续套图和导出流程使用。
- 页面内容支持设置中的中文/英文切换。

## 10. 通用规则

1. 所有 API Key 必须放在环境变量。
2. 前端不能暴露 service role key、图片模型 key 或 SDS key。
3. 所有 AI 调用必须走后端接口。
4. 所有批量任务必须记录总数、成功数、失败数和失败原因。
5. 所有图片处理结果必须保留原图记录。
6. 所有生成文件必须上传到 Supabase Storage，不写入 Vercel 的 `public` 目录。
7. 新增功能必须补充验收标准。
8. 不确定业务规则时，先补充“待确认问题”，不要直接默认实现。

## 11. 非功能需求

| 文件存储 | 生成文件统一存储到 Supabase Storage |
| 质量检查 | 每次代码改动后运行 `npm run lint` 和 `npm run build` |

## 15. 当前重点测试场景

1. 上传 10 张图片。
